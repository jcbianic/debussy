## Summary

Foundation setup establishes the complete development ecosystem for Debussy. This includes:
- **Developer Experience**: Single-command dev server with hot reloading (FR-100) and clear test/build commands (FR-101, FR-102)
- **Quality Gates**: Automated testing framework with coverage reporting (FR-102, FR-120)
- **Global Distribution**: npm package configuration and `npx debussy` executable support (FR-110, FR-111)
- **Visual Polish**: Design tokens system for theming (FR-200), welcome page with navigation (FR-300, FR-301), and integrated core pages (FR-302)

The foundation delivers six polished, interconnected user experiences: developer quick start, test execution, package distribution, customizable appearance, welcoming first impression, and unified navigation.

## Design Decisions

### 1. Developer Experience (FR-100)

**Decision**: Use `npm run dev` as the primary dev command.

**How it works**:
- `npm run dev` invokes `nuxt dev` via npm scripts
- Nuxt automatically enables hot module reloading (HMR)
- Dev server accessible at `http://localhost:3000` (configurable)
- No environment setup required; dependencies handle configuration

**Why**: Nuxt's dev server is mature, well-documented, and handles HMR out of the box. npm scripts are the standard entry point for all Node.js projects.

### 2. Testing Framework (FR-102, FR-120)

**Decision**: Vitest 3.2 as primary test runner.

**How it works**:
- `npm test` runs Vitest in watch mode
- Tests are co-located with source (`*.test.ts` / `*.spec.ts`)
- Vitest discovers tests automatically via glob pattern
- Test output includes coverage report (--coverage flag)
- Integration and E2E tests use Vitest's mock/spy utilities

**Why**: Vitest is fast, Vue-aware, and has minimal configuration. Test discovery is automatic, reducing boilerplate.

### 3. Theming System (FR-200)

**Decision**: CSS variables + Tailwind design tokens + Vue composable for runtime switching.

**How it works**:
- Define design tokens in `theme.ts` (colors, typography, spacing)
- Tailwind CSS uses these tokens as config values
- CSS variables in `:root` and `[data-theme="dark"]` selector
- `useTheme()` composable provides reactive theme switching
- Theme persists to localStorage; preference synced on mount

**Why**: CSS variables enable instant theme switching without page reload. Tailwind ensures consistency across components. Composable allows any component to access/change theme reactively.

### 4. Package Distribution (FR-110, FR-111)

**Decision**: npm package with `bin` field pointing to Nitro standalone executable.

**How it works**:
- `package.json` includes `"bin": { "debussy": "./bin/debussy.cjs" }`
- `npm publish` uploads to npm registry
- Users run `npx debussy` → npm downloads latest, executes bin entry
- Bin entry is a Nitro standalone build (no Node modules required in user env)
- Executable starts server, opens browser to localhost

**Why**: npm distribution is the standard for Node.js CLI tools. Standalone Nitro build eliminates node_modules overhead and makes distribution lightweight.

### 5. Navigation & Pages (FR-301, FR-302)

**Decision**: AppNavigation component (header + sidebar) shared across all pages; file-based routing via Nuxt.

**How it works**:
- `pages/index.vue` = welcome page
- `pages/sessions.vue` = session viewer
- `pages/workflows.vue` = workflow viewer
- `pages/artifacts.vue` = artifact browser
- AppNavigation appears on all pages (via layout)
- Active page highlighted via `$route.name` reactive property
- Mobile-responsive nav with hamburger menu (Radix Vue)

**Why**: Nuxt file-based routing is zero-config. Shared layout component ensures consistency. Radix Vue provides accessible, pre-styled primitives.

### 6. Welcome Page (FR-300)

**Decision**: Hero section + feature callouts + dual CTAs.

**How it works**:
- Hero headline: "Orchestrate Claude Code sessions locally"
- Three feature callouts: Sessions, Workflows, Artifacts
- Primary CTA: "New Session" → `/sessions`
- Secondary CTA: "View Workflows" → `/workflows`
- Responsive grid layout (mobile stacks, desktop flexes)

**Why**: Clear value proposition reduces abandonment. Prominent CTAs guide new users to main features.

### 7. Test Patterns (FR-120, FR-121)

**Decision**: Three tiers of tests: unit (components/utils), integration (API routes), E2E (user flows).

**How it works**:
- **Unit tests**: Each component/utility has a `.test.ts` file
- **Integration tests**: API routes tested with Nitro test utilities
- **E2E tests**: Full user workflows tested via jsdom or Playwright
- All tests co-located with source; Vitest auto-discovers

**Why**: Multi-tier testing provides confidence at different levels. Co-location keeps tests close to the code they validate. Auto-discovery reduces friction.

### 8. Build & Production (FR-101)

**Decision**: `npm run build` produces optimized Nuxt SSR + static assets + Nitro server.

**How it works**:
- `nuxt build` compiles TypeScript, bundles components, optimizes assets
- Output: `.output/server/` (Nitro server), `.output/public/` (static files)
- Standalone binary can be deployed or run via `npm start`
- No build steps required for `npx debussy` — uses pre-built binary

**Why**: Nuxt's build process is mature and handles code splitting, minification, and asset optimization. Standalone output allows distribution without build tools.

## Data Model

The foundation phase does not persist complex state. However, sessions and workflows will store minimal metadata:

```typescript
// Session model (SQLite)
interface Session {
  id: string                    // UUID
  createdAt: Date              // Session start
  label?: string               // User-given name
  status: "idle" | "running" | "complete"
}

// Workflow model (SQLite)
interface Workflow {
  id: string                    // UUID
  sessionId: string            // FK → Session
  phase: string                // e.g., "specify", "plan", "implement"
  status: "pending" | "running" | "complete"
  output: string               // Aggregated stdout/stderr
}

// Theme model (localStorage)
interface ThemePreference {
  mode: "light" | "dark" | "system"
  customPrimary?: string       // Hex color override
  customSecondary?: string
}
```

- Mock child components where needed

### Integration Tests

- Theme API tested with storage mocking

### E2E Tests

- User clicks "New Session" → navigates to sessions page
- User creates session → workflow appears in workflows page
- User switches theme → colors update (no reload)

### Success Metrics

## Deployment & Distribution

1. **Local Development**: `npm run dev` (5 mins to first run)
2. **Pre-publish Testing**: `npm test && npm run build`
3. **npm Publish**: `npm publish` (requires npm auth)
4. **End-user Execution**: `npx debussy` (auto-downloads, runs)
5. **Updates**: `npm install -g debussy@latest` or auto-checked via npx

## Risks & Mitigation

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|-----------|
| Slow dev server startup | Medium | Blocks flow | Nuxt's lazy loading; HMR caching |
| Test flakiness | Medium | Reduces confidence | Deterministic mocks; async handling |
| Theme system complexity | Low | Over-engineering | CSS variables only; no Emotion/styled-components |
| npm publish failures | Low | Blocks distribution | Local testing of bin entry; CI/CD validation |
| Mobile layout issues | Medium | Poor UX on phones | Responsive grid + Tailwind breakpoints |
